# Story 2.5: Remote Control and Responsive Polish

## Status
Approved

## Story
**As a** user,
**I want** the remote control view and responsive breakpoints to follow the same glassmorphism design, and accessibility/motion preferences to be respected,
**so that** the experience is consistent across all contexts and accessible to all users.

## Acceptance Criteria
1. `public/skins/default/remote.css` rewritten with forest-green glassmorphism: translucent surfaces, white-at-opacity borders, green accents, tight layout
2. Remote panel controls (play/pause, slider readouts) styled consistently with main UI components from Story 2.3
3. Responsive behavior: panels reflow cleanly at mobile widths (below 768px), maintaining translucent surfaces and readability -- no layout animations during reflow
4. Projection mode margins (`--projection-margin-*`) remain functional with the new design
5. `@media (prefers-reduced-motion: reduce)`: disables modal scale transition (use `opacity` fade only), disables any green glow pulse animations, disables hover glow transition
6. `@media (prefers-contrast: more)`: bumps border opacity from `0.1` to `0.2`, text-muted opacity from `0.5` to `0.7`, surface-card backgrounds gain +0.02 opacity for better distinction
7. No new hue families introduced -- stays achromatic + green with narrow semantic exceptions (amber, red, blue, purple for status only)
8. All colors in oklch -- no hex/hsl in remote.css

## Dev Notes

### Current Remote CSS
- `public/skins/default/remote.css` provides styling for the remote control interface
- Remote view is a simplified panel view accessed from another device (phone/tablet)
- It connects via WebSocket and shows panel controls without the code editors

### Key Files
- `public/skins/default/remote.css` (full rewrite)
- `public/skins/default/theme.css` (add media queries here or in separate block)
- `static/css/base.css` (responsive rules may exist here)
- `static/css/accessibility.css` (existing accessibility styles -- may need updates)

### Responsive Breakpoints (current)
The existing CSS doesn't have formal breakpoints documented. Common approach:
- `< 768px`: mobile (single column, stacked panels)
- `768px - 1024px`: tablet (narrower panels, 2-column possible)
- `> 1024px`: desktop (full layout, multiple columns)

### Projection Mode
Projection mode adds margins around the content area for screen sharing (avoiding OBS/streaming overlays). It uses CSS custom properties:
```css
--projection-margin-top: 0px;
--projection-margin-right: 0px;
--projection-margin-bottom: 0px;
--projection-margin-left: 0px;
```
These are applied to the main content container. The background gradient should extend full-viewport, not just within the margins.

### Technical Constraints
- `prefers-reduced-motion` is well-supported in all target browsers/Electron
- `prefers-contrast` is supported in Chrome 96+, Firefox 101+, Safari 14.1+ -- Electron 39 supports it
- Remote CSS must work standalone (loaded separately from main theme when in remote view)
- Remote view may not load `variables.css` directly -- check if CSS variables are available in remote context, or if remote.css needs to define its own

### Accessibility Notes
- Existing `static/css/accessibility.css` has focus indicator styles and screen reader utilities
- This story should NOT modify focus indicators (they're functional, not aesthetic)
- Reduced motion and high contrast are additive improvements on top of the base design

## Testing
- Open remote control view in a mobile browser: verify glassmorphism styling matches main UI
- Resize browser window through breakpoints: verify panels reflow without layout animations
- Enable "Reduce motion" in OS settings: verify modal fades instead of scales, no glow animations
- Enable "Increase contrast" in OS settings: verify borders and text become more visible
- Set projection margins in settings: verify content shifts with gradient background behind margins
- Verify no new color hues appear anywhere in the UI
- Final sweep: grep all skin CSS files for hex/hsl values
