# Story 3.2: electron-builder Major Version Upgrade (25 -> 26)

## Status
Ready for Review

## Story
**As a** maintainer,
**I want** `electron-builder` upgraded from 25.1.8 to 26.x,
**so that** the 5 high-severity transitive vulnerabilities (`minimatch`, `tar` x4, `@electron/asar`, `@electron/rebuild`) are resolved and the CI build pipeline uses a supported builder version.

## Acceptance Criteria
1. `electron-builder` updated from `25.1.8` to latest `26.x` in `package.json`
2. The following transitive advisories are resolved:
   - `minimatch` ReDoS (GHSA-3ppc-4f35-3m26, HIGH)
   - `tar` race condition (GHSA-r6q2-hw4h-h46w, HIGH)
   - `tar` hardlink path traversal (GHSA-34x7-hfp2-rc4v, HIGH)
   - `tar` symlink poisoning (GHSA-8qq5-rm4j-mr97, HIGH)
   - `tar` hardlink escape via symlink chain (GHSA-83g3-92jg-28cx, HIGH)
3. `electron-builder.json` config remains functional -- adjust any deprecated or renamed config keys for v26
4. Local dev build succeeds: `npm run build` produces working Electron app
5. CI workflow `.github/workflows/build-electron.yml` (Story 1.2) works with updated builder -- all three platform builds pass
6. macOS code signing still works (CSC_LINK / CSC_KEY_PASSWORD flow unchanged or adapted)
7. Artifact output structure matches Story 1.2 expectations: dmg, zip (macOS), exe (Windows), AppImage, deb (Linux)
8. `npm audit` shows zero fixable high/critical vulnerabilities after this upgrade

### Rollback Plan
If v26 introduces unfixable issues:
1. Revert `package.json` and `package-lock.json` to pre-upgrade state
2. The vulnerabilities are in build-time transitive deps (`tar`, `minimatch`) -- not in the shipped application runtime
3. Risk acceptance: these are build-tool vulnerabilities, not user-facing -- acceptable to defer if v26 is broken

### Risk Assessment
- **Primary Risk**: electron-builder v26 config incompatibility breaks CI builds
- **Mitigation**: Test locally first, review v26 changelog, keep old config as reference
- **Impact if deferred**: Build-time-only vulnerability -- does not affect end users running the app
- **Severity context**: The `tar` and `minimatch` vulns are in build tooling, not shipped code. ReDoS in `minimatch` requires crafted glob patterns (not user-controlled in CI). `tar` path traversal requires malicious archive (not applicable in controlled CI).
