# Story 1.5: Orchestrator Workflow

## Status
Done

## Story
**As a** maintainer,
**I want** a single top-level workflow that orchestrates version detection, Electron builds, release creation, and pages deployment based on push context,
**so that** one push to `main` triggers the appropriate pipeline without me managing multiple workflows manually.

## Acceptance Criteria
1. `.github/workflows/ci.yml` triggers on push to `main` and on `workflow_dispatch` with optional version input
2. Calls detect-version workflow (Story 1.1) as first job
3. If `is_release == true`: calls build-electron workflow (Story 1.2), then runs release creation job (Story 1.3)
4. Always calls deploy-pages workflow (Story 1.4) regardless of release status -- docs deploy on every push
5. Pages deployment and Electron build run in parallel (they are independent)
6. Old workflow files removed: `release.yml`, old `deploy-pages.yml`, `docs.yml.disabled`, `static.yml.disabled`
7. Manual `workflow_dispatch` allows triggering a release build by providing a version string

## Dev Notes

### Current State
- Two independent workflows: `release.yml` (release builds) and `deploy-pages.yml` (pages)
- No orchestration -- a release commit triggers both independently
- Pages deploy runs on every push to `main` but release only on `^^^` pattern
- This story replaces both with a single orchestrated pipeline

### Key Files
- `.github/workflows/ci.yml` (new -- the orchestrator)
- `.github/workflows/detect-version.yml` (Story 1.1)
- `.github/workflows/build-electron.yml` (Story 1.2)
- `.github/workflows/deploy-pages.yml` (Story 1.4 -- reusable version)
- `.github/workflows/release.yml` (delete)
- `.github/workflows/docs.yml.disabled` (delete)
- `.github/workflows/static.yml.disabled` (delete)

### Workflow Graph
```
ci.yml (push to main)
  |
  +---> detect-version.yml
  |       |
  |       +--[is_release=true]--> build-electron.yml
  |       |                         |
  |       |                         +---> create-release (job)
  |       |
  |       +--[is_release=false]--> (skip builds)
  |
  +---> deploy-pages.yml (always, parallel with above)
```

### Technical Constraints
- Reusable workflows called with `uses:` must reference the workflow file path relative to the repo
- Jobs using `needs:` create sequential dependencies; jobs without `needs:` run in parallel
- Conditional job execution: `if: needs.detect-version.outputs.is_release == 'true'`
- Reusable workflows inherit secrets with `secrets: inherit` or explicit secret passing
- Maximum workflow depth: caller -> reusable is one level (no nested reusable workflows calling other reusable workflows in GitHub Actions -- but the orchestrator calling reusable workflows is fine)

### Permissions
```yaml
permissions:
  contents: write    # For tag creation and release
  pages: write       # For Pages deployment
  id-token: write    # For Pages OIDC
```
