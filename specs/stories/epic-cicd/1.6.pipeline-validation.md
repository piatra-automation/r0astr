# Story 1.6: Pipeline Validation

## Status
Approved

## Story
**As a** maintainer,
**I want** to validate the entire CI/CD pipeline end-to-end with test releases,
**so that** I have confidence the new pipeline works reliably before depending on it for production releases.

## Acceptance Criteria
1. Push a non-release commit to `main` -- verify only pages deployment runs, no release jobs triggered
2. Push a `^^^ v0.10.16-rc.1` commit -- verify full pipeline: tag creation, all three platform builds, prerelease GitHub Release with artifacts, and pages deployment
3. All platform artifacts (dmg, zip, exe, AppImage, deb) build successfully and attach to the GitHub Release
4. GitHub Pages site deploys with updated docs at root and lite app at `/app/`
5. Old workflow files (`release.yml`, `*.yml.disabled`) are confirmed deleted from the repository
6. Release process is documented in README.md under a "Releases" section (3-5 lines covering the `^^^ vX.Y.Z` pattern, what triggers, and manual workflow_dispatch option)

## Tasks / Subtasks
- [ ] **Task 1: Non-release push validation** (AC: 1)
  - [ ] Push a minor docs or code change to `main`
  - [ ] Verify `ci.yml` runs, detect-version outputs `is_release=false`

  - [ ] Verify all three platform build jobs complete

  - [ ] Browse `r0astr.org` and verify MkDocs site loads
  - [ ] Browse `r0astr.org/app/` and verify lite app loads

  - [ ] Confirm `.github/workflows/release.yml` is absent

## Dev Notes

### Validation Checklist
Run through this matrix:

| Scenario | Expected Result |
|----------|----------------|
| Push without `^^^` | Pages deploy only |
| Push with `^^^ v0.10.16-rc.1` | Full pipeline + prerelease |
| Manual workflow_dispatch `v0.10.17-test.1` | Full pipeline + prerelease |
| Push with `^^^ v0.11.0` (clean version) | Full pipeline, NOT prerelease |

### Key Files
- `README.md` (add release docs section)
- All `.github/workflows/` files (verify final state)

### Technical Notes
- The `-rc.1` prerelease test avoids polluting the version numbering for the next real release
- If any platform build fails, the release job should not run (it `needs:` all build jobs)
- After validation, the `-rc.1` release can be deleted manually from GitHub if desired
- Pages deployment is independent and should succeed even if release builds fail

### Rollback Plan
If the new pipeline fails:
1. The old workflow files are in git history and can be restored
2. Manual `gh release create` can be used as emergency fallback
3. Pages can be deployed manually with `mkdocs gh-deploy`
